TABLE SUMMARY
================================================================================

buddy_listings (5 fields + metadata)
  Purpose: Dive buddy postings
  Records: Typically 100-10K active at any time
  Key Access: Browse by status/location/date
  Privacy: Contact info not stored

buddy_interests (6 fields)
  Purpose: Match requests between divers
  Records: Proportional to listings × engagement
  Key Access: By listing_id or interested_user_id
  Status Flow: pending → accepted → connected

buddy_connections (8 fields)
  Purpose: Approved buddy pairs
  Records: Grows with successful matches
  Key Access: By user_id (normalized)
  Safety: UNIQUE constraint prevents duplicates

buddy_messages (7 fields)
  Purpose: Direct messages between buddies
  Records: Grows with engagement
  Key Access: By sender/receiver
  Security: Connection required to send

================================================================================
CONTACT INFO PROTECTION MECHANISM
================================================================================

Design:
- NOT stored in buddy_listings table
- Stored in auth.users.raw_user_meta_data (encrypted by Supabase)
- Revealed via get_buddy_profile() function
- Tracked via contact_info_revealed_at timestamp

Visibility States:
1. Browsing: Contact info hidden (RLS enforces)
2. Interest Pending: Contact info still hidden
3. Interest Accepted: Contact info REVEALED (connection created)
4. Connected: Full contact info visible
5. Own Profile: Always visible

Implementation:
- get_buddy_profile(user_id) checks for valid connection
- Returns NULL email/phone if not authorized
- Database checks connection existence
- RLS enforces at table level

Threat: User A sees User B's email
Mitigation: RLS policy + get_buddy_profile() function

Threat: User A modifies User B's listing
Mitigation: UPDATE RLS policy checks user_id = auth.uid()
Status: ✓ PROTECTED

Threat: User A deletes User B's listing
Mitigation: DELETE RLS policy checks user_id = auth.uid()
Status: ✓ PROTECTED

Threat: User A expresses interest in own listing
Mitigation: Database CHECK constraint
Status: ✓ PROTECTED

Threat: Duplicate interests created
Mitigation: UNIQUE constraint (listing_id, interested_user_id)
Status: ✓ PROTECTED

Threat: Duplicate connections created
Mitigation: UNIQUE constraint (user_id_1, user_id_2)
Status: ✓ PROTECTED

Threat: User A sends message to non-connected User B
Mitigation: RLS policy checks buddy_connections EXISTS
Status: ✓ PROTECTED

Threat: Anonymous user browses listings
Mitigation: RLS policy requires auth.uid() IS NOT NULL
Status: ✓ PROTECTED

================================================================================
STORED PROCEDURES
================================================================================

1. accept_buddy_interest(interest_id)
   - Validates calling user is listing owner
   - Updates interest status to 'accepted'
   - Sets contact_info_revealed_at = NOW()
   - Creates buddy_connection (normalized user IDs)
   - Returns connection details
   - SECURITY DEFINER for integrity

2. reject_buddy_interest(interest_id)
   - Updates interest status to 'rejected'
   - Only listing owner can reject
   - Removes interest from queue

3. get_buddy_profile(user_id)
   - Returns user profile with conditional fields
   - Email/phone NULL if not connected
   - Shows/hides based on connection status
   - Safe RLS implementation

4. auto_expire_listings()
   - Trigger function
   - Auto-transitions to 'expired' when due
   - Runs BEFORE INSERT/UPDATE

5. update_timestamp()
   - Trigger function
   - Sets updated_at = NOW()
   - Runs BEFORE UPDATE
