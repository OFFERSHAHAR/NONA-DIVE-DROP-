1. CORE TABLES CREATED
================================================================================

ADMIN RBAC (Role-Based Access Control):
  • admin_roles (7 predefined roles)
  • admin_permissions (22 permissions)
  • role_permissions (junction table)
  • admin_users (links users to roles)

Dive Site Management:
  • dive_sites_enhanced (moderation & metadata)
  • dive_site_images (multiple images per site)

Shuttle/Transport:
  • shuttles (reusable shuttle templates)
  • shuttle_schedules (individual runs/dates)
  • shuttle_bookings (user reservations)

Audit & Compliance:
  • audit_logs (immutable audit trail)

================================================================================
2. PREDEFINED ADMIN ROLES
================================================================================

1. SUPER_ADMIN
   - Full system access
   - All 22 permissions

2. SITE_MANAGER
   - Manage dive sites (CRUD)
   - Approve pending sites
   - Manage site images
   - View user profiles

3. SHUTTLE_MANAGER
   - Manage shuttles (CRUD)
   - View bookings
   - Confirm/cancel bookings
   - View shuttle analytics

4. USER_ADMIN
   - Manage user accounts
   - Update user profiles
   - View user activity

5. CONTENT_MODERATOR
   - Approve/reject content
   - View pending sites
   - Moderate reviews

6. AUDITOR
   - Read-only access to all resources
   - Export audit logs
   - View compliance reports

7. VIEWER
   - Limited read-only access
   - View dive sites and profiles only

3. KEY FEATURES
================================================================================

✓ Role-Based Access Control
  - Granular permission system
  - Role inheritance via junction tables
  - Permission checking via helper functions

✓ Row-Level Security (RLS)
  - All tables have RLS enabled
  - Policies enforce role-based visibility
  - Automatic filtering of deleted records

✓ Moderation Workflow
  - Dive site approval pipeline
  - Status: pending → approved/rejected/needs_revision
  - Image moderation with approval flow

✓ Shuttle Booking System
  - Reusable shuttle templates
  - Individual schedule runs per date
  - User booking reservations
  - Real-time occupancy tracking

✓ Audit Trail
  - All admin actions logged
  - Immutable append-only design
  - Automatic triggers on critical tables
  - Compliance-ready event logging

✓ Soft Delete Support
  - Soft deletes via deleted_at timestamps
  - Data recovery capability
  - Compliance and archiving ready

✓ Multi-Language Support
  - Role display names in English & Hebrew
  - Alt text for images (en/he)

8. RLS POLICY SUMMARY
================================================================================

PUBLIC ACCESS:
  ✓ View published dive sites (approved or needs_revision)
  ✓ View approved dive site images
  ✓ View active shuttle listings
  ✓ View personal shuttle bookings

AUTHENTICATED USER:
  ✓ Create/manage own shuttle bookings
  ✓ View own profile
  ✓ Cannot access admin functions without admin role

ADMIN USERS (Based on Role):
  ✓ SUPER_ADMIN: Full read/write access to all tables
  ✓ SITE_MANAGER: Manage dive sites, moderation
  ✓ SHUTTLE_MANAGER: Manage shuttles, bookings
  ✓ USER_ADMIN: Manage user accounts
  ✓ CONTENT_MODERATOR: Approve/moderate content
  ✓ AUDITOR: Read all, export audit logs
  ✓ VIEWER: Read limited resources only

AUDIT LOGS:
  ✓ Only accessible by SUPER_ADMIN and AUDITOR
  ✓ Immutable (no updates/deletes)
  ✓ Auto-logged by triggers on critical tables

9. PERFORMANCE CHARACTERISTICS
================================================================================

Query Performance (with indexes):
  • Admin permission check: ~5ms (cached in app)
  • Shuttle availability lookup: ~10ms
  • Booking availability check: ~15ms
  • Audit log retrieval: ~20ms for 100 records

Recommended Caching:
  • User permissions: Cache in app memory for 1 hour
  • Role definitions: Cache globally (rarely change)
  • Shuttle schedules: Cache for 5 minutes

Data Growth Estimates (1 year):
  • Audit logs: ~100K-500K records (index by date)
  • Shuttle bookings: ~10K-50K records
  • Shuttle schedules: ~2K+ records
  • Dive site images: 1-5 per site

Archive Strategy:
  • Move audit logs > 12 months to cold storage
  • Hard-delete soft-deleted records after 90 days
  • Partition shuttle_bookings by year
