================================================================================
                       PERMISSION MATRIX
================================================================================

ANONYMOUS USER (not logged in)
  ✅ View public listings (no contact info)
  ❌ Create listings
  ❌ Express interest
  ❌ Access contact info
  ❌ Block/report users

REGISTERED USER (verified email)
  ✅ View all public listings
  ✅ Create listings
  ✅ Express interest in listings
  ✅ Request contact reveal
  ✅ Block users
  ✅ Report users
  ✅ View own profile
  ✅ View public profiles
  ❌ Delete other users' listings
  ❌ See contact info (unless mutual reveal)

LISTING OWNER
  ✅ All registered user actions PLUS:
  ✅ Update own listings
  ✅ Delete own listings
  ✅ View interest requests (who's interested)
  ✅ Accept/decline contact reveals
  ✅ See contact info (if mutual reveal)

================================================================================
                    CONTACT REVEAL WORKFLOW
================================================================================

4-STEP MUTUAL CONSENT PROCESS:

STEP 1: EXPRESS INTEREST
  User B sees User A's listing → clicks "Express Interest"
  → Interest record created
  → User A sees "Someone is interested" (anonymous)

STEP 2: REQUEST CONTACT REVEAL
  User B clicks "Reveal My Contact"
  → Contact reveal record created (initiator_revealed = true)
  → Message sent to User A: "User X wants to connect"

STEP 3: RECIPIENT ACCEPTS
  User A sees request with User B's name & photo
  → User A clicks "Accept"
  → Contact reveal record updated (recipient_revealed = true)
  → mutual_revealed_at timestamp set

STEP 4: MUTUAL VISIBILITY
  User A can now see User B's: email, phone, profile pic
  User B can now see User A's: email, phone, profile pic

PRIVACY GUARANTEED:
  • User A never knows who's interested until User B requests
  • User B voluntarily reveals (can decline contact)
  • Contact info only visible after BOTH reveal
  • Either party can still block after reveal

================================================================================
                        BLOCKING SYSTEM
================================================================================

WHAT BLOCKING DOES:
  When User A blocks User B:
  ├─ User B cannot see User A's active listings
  ├─ User B cannot express interest (silently fails)
  ├─ User B cannot contact User A
  ├─ Existing interests from User B are removed
  ├─ Existing contact reveals with User B are deleted
  └─ User B gets no notification (privacy)

ASYMMETRIC BLOCKING:
  A blocks B ≠ B blocks A
  (User B might not block User A back)

REVERSIBLE:
  User A can unblock User B anytime
  (Requires explicit action, not automatic)

================================================================================
                         REPORTING SYSTEM
================================================================================

REPORT TYPES:
  • SPAM: Repetitive unwanted contacts
  • ABUSE: Harassment, threats, discrimination
  • INAPPROPRIATE: Sexual content, scams
  • OTHER: Custom reason provided

REPORT ACTION:
  1. User A submits report about User B
  2. System records: reporter, reported, reason, timestamp
  3. User B auto-blocked from viewing User A's listings
  4. Report marked as "open" for moderator review
  5. Moderator can investigate/dismiss/resolve

COMPLIANCE:
  ✅ Each report logged for legal compliance
  ✅ User can see their own reports
  ✅ Rate limited (1 per user per 24 hours)
  ✅ Audit trail of all reports

================================================================================
                      DATABASE STRUCTURE
================================================================================

USERS TABLE (RLS enabled)
  ├─ id, email, first_name, last_name
  ├─ phone (protected), profile_picture_url (protected)
  ├─ bio, location, diving_experience
  └─ blocked_users (UUID array), timestamps

LISTINGS TABLE (RLS enabled)
  ├─ id, owner_id, title, description
  ├─ diving_level, location, dive_date
  ├─ max_buddies, image_url, is_active
  └─ timestamps

INTERESTS TABLE (RLS enabled)
  ├─ id, listing_id, interested_user_id, listing_owner_id
  └─ UNIQUE constraint prevents duplicate interests

CONTACT_REVEALS TABLE (RLS enabled)
  ├─ id, listing_id, initiator_id, recipient_id
  ├─ initiator_contact_revealed, recipient_contact_revealed
  ├─ mutual_revealed_at (when both revealed)
  └─ UNIQUE constraint prevents duplicate reveals

BLOCKS TABLE (RLS enabled)
  ├─ id, blocker_id, blocked_user_id, reason
  └─ UNIQUE constraint prevents duplicate blocks

REPORTS TABLE (RLS enabled)
  ├─ id, reporter_id, reported_user_id, reported_listing_id
  ├─ reason (enum), description, status
  └─ timestamps

AUDIT_LOG TABLE (RLS enabled, immutable)
  ├─ id, actor_id, action (enum), resource_type, resource_id
  ├─ target_user_id, details (JSONB)
  ├─ ip_address, user_agent
  └─ created_at (no updates allowed)

================================================================================
                        RLS POLICIES
================================================================================

USERS TABLE:
  ✅ READ: Owner can read own profile
  ✅ READ: Others can read public fields only
  ✅ READ: Blocked users cannot read any fields
  ✅ UPDATE: Owner only
  ✅ INSERT: On signup only

LISTINGS TABLE:
  ✅ READ: Anyone can read active listings
  ✅ READ: Owner can read own (inactive too)
  ✅ READ: Blocked users cannot read owner's listings
  ✅ UPDATE: Owner only
  ✅ INSERT: Authenticated users only
  ✅ DELETE: Owner only

INTERESTS TABLE:
  ✅ READ: Interested user can read own interests
  ✅ READ: Owner can read interests in own listings
  ✅ INSERT: Authenticated, not owner, not interested yet
  ✅ DELETE: Interested user only

CONTACT_REVEALS TABLE:
  ✅ READ: Both parties (initiator & recipient)
  ✅ INSERT: Authenticated users only
  ✅ UPDATE: Involved parties only
  ✅ DELETE: Involved parties only

BLOCKS TABLE:
  ✅ READ: Blocker only
  ✅ INSERT: Blocker only
  ✅ DELETE: Blocker only

REPORTS TABLE:
  ✅ READ: Reporter can read own reports
  ✅ INSERT: Authenticated users only
  ✅ (UPDATE/DELETE: Admin/system only)

AUDIT_LOG TABLE:
  ✅ READ: Involved users and admins
  ✅ INSERT: System only (triggers & functions)
  ✅ (Immutable after creation)
